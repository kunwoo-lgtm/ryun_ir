# 게런티 함수 기반 블록체인 수익배분 (RYUN Guarantee Distribution)

음성 메모(2026-07-22)에서 정리된 원칙을 블록체인 수익배분 함수로 구현한 것입니다.

## 음성 메모의 요구사항 → 설계 반영

| 음성 메모 내용 | 설계 반영 |
|---|---|
| 하청 구조를 파트너 간 수익 분배 공식으로 전환 | 파트너 등록제 — 발주/하청이 아닌 동등한 분배 대상 |
| 예산이 적어 게런티를 다 못 줄 때가 있다 | 예산 부족 시 게런티 비례 배분, **부족분은 체인에 누적(arrears)되어 다음 회차 최우선 상환** |
| 함수로 정하면 눈에 보이니까 아무도 불만이 없다 | 분배 규칙 전체가 스마트 컨트랙트 코드로 공개, 모든 지급이 이벤트로 기록 |
| N값만 조정하면 된다 | 조정 가능한 것은 파라미터뿐: 게런티 금액 `G`, 가중치 `w`(N값), 수수료율 |
| 유통(매출)이 커지면 분배도 커진다 | 게런티 충족 후 초과 수익을 가중치 비례로 전액 배분 |
| 기존 티켓팅 플랫폼은 독점·고수수료 | 플랫폼 수수료 상한 5%(`MAX_FEE_BPS`) 코드 강제 |

## 게런티 함수 정의

회차 수익 `R`, 수수료율 `f` → 순예산 `B = R × (1 − f)`

```
1단계  누적 미지급 게런티(A_i) 우선 상환
       B < ΣA 이면 A_i 비례 배분, 잔여분은 계속 누적
2단계  당회차 게런티(G_i) 지급
       잔여예산 < ΣG 이면 G_i 비례 배분, 부족분은 A_i 에 누적
3단계  초과 수익 S 를 가중치 w_i(N값) 비례로 전액 배분
       지급액_i = S × w_i / Σw
```

예산이 부족한 회차에도 게런티 채권이 소멸하지 않고 온체인에 남으므로,
"이번엔 못 줬다"가 아니라 "언제 얼마가 밀려 있고 언제 상환됐는지"가 모두에게 보입니다.

## 파일

- `RyunGuaranteeDistribution.sol` — Solidity 0.8.20 스마트 컨트랙트 (solc 컴파일 검증 완료)
  - `settle()` : 수익 입금 + 회차 정산 (게런티 함수 실행)
  - `withdraw()` : 파트너 인출 (pull-payment)
  - `setPartner(주소, 게런티, 가중치)` : 파라미터 조정
  - `partnerInfo(주소)` : 게런티·가중치·미지급 누적·잔고 공개 조회
- `guarantee-distribution.mjs` — 동일 규칙의 JS 레퍼런스 구현 + 시뮬레이션

## 시뮬레이션 실행

```bash
node contracts/guarantee-distribution.mjs
```

파트너 3인(아티스트A 게런티 50만/가중치 3, 아티스트B 30만/2, 기획·운영 20만/5), 수수료 3% 기준:

- **회차 1 (수익 60만, 예산 부족)** : 게런티 비례 배분, 총 41.8만 원 미지급 누적
- **회차 2 (수익 130만)** : 밀린 41.8만 원 전액 우선 상환 → 당회차 게런티 84.3만 비례 지급
- **회차 3 (수익 300만, 유통 확대)** : 미지급 완전 해소 + 게런티 전액 + 초과 175.3만 원 가중치 배분
